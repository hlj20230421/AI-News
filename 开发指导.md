# AI 资讯智能聚合与推送系统 分步开发指导

> 配套文档：`需求文档.md`  
> 编写日期：2026-04-23  
> 适用读者：独立开发者 / 小团队（1-2 人）  
> 预计总工期：8 周（可压缩至 4 周，功能取舍见附录）

---

## 使用说明

本指导按 **Step 0 → Step 4** 推进，每一步包含：

- 🎯 **目标**：这一步完成后的产出状态
- 📦 **交付物**：可运行/可验证的代码或配置
- 📋 **任务拆解**：编号子任务（可直接作为 TODO 清单）
- ✅ **验收标准**：如何确认完成
- ⚠️ **常见坑**：提前避雷

推荐工作节奏：**一次只聚焦一个 Step，Step 内部按任务编号顺序推进**，每完成一个任务就 commit 一次。

## Step 0：准备与项目骨架（第 0 周，1-2 天）

### 🎯 目标

搭好"地基"，让后续任意模块可以立刻开始编码，不被环境问题阻塞。

### 📦 交付物

- 一个可运行 `docker compose up` 的项目骨架
- 代码仓库 + 基础 CI
- 所有第三方凭证就绪

### 📋 任务拆解

**0.1 账号与 Key 准备**

- [ ] 注册 / 准备一个 **LLM API Key**（优先 OpenAI；备选 Anthropic / DeepSeek / 月之暗面 / 本地 Ollama）
- [ ] 注册 **飞书开放平台**，创建**自建应用**或直接建一个**群机器人（Webhook）**
  - 群机器人最简：群聊 → 设置 → 群机器人 → 添加"自定义机器人" → 记下 Webhook URL + 签名密钥
- [ ] （可选）准备 **GitHub Token**（提升 GitHub API 配额到 5000/h）
- [ ] （可选）准备代理（部分信息源需科学上网）

**0.2 代码仓库初始化**

- [ ] 新建 Git 仓库（GitHub/Gitee/自建）
- [ ] 建立如下目录结构：

```
ai-news/
├── docker-compose.yml
├── .env.example
├── .gitignore
├── README.md
├── pyproject.toml          # 或 requirements.txt
├── backend/
│   ├── app/
│   │   ├── __init__.py
│   │   ├── main.py         # FastAPI 入口
│   │   ├── config.py       # 配置加载
│   │   ├── db/             # SQLAlchemy 模型 + session
│   │   ├── collectors/     # 采集器
│   │   ├── analyzers/      # LLM 分析
│   │   ├── notifiers/      # 推送通道
│   │   ├── scheduler/      # Celery tasks + beat
│   │   ├── api/            # REST 路由
│   │   └── utils/
│   └── tests/
├── frontend/               # Step 2 才启用，先占位
├── scripts/                # 数据迁移、一次性脚本
├── ops/
│   ├── Dockerfile.backend
│   ├── Dockerfile.frontend
│   └── grafana/            # Step 4 启用
└── docs/
    ├── 需求文档.md
    └── 开发指导.md
```

**0.3 基础依赖与配置**

- [ ] 使用 **uv** 或 **poetry** 管理 Python 依赖（Python 3.11+）
- [ ] 安装核心包：`fastapi uvicorn sqlalchemy psycopg[binary] alembic celery redis feedparser trafilatura httpx pydantic-settings python-dotenv openai litellm`
- [ ] 写 `.env.example`（包含所有环境变量占位，见需求文档附录 C）
- [ ] 写 `docker-compose.yml`，至少包含：`postgres（含 pgvector）` + `redis` + `backend` + `worker` + `beat`

**0.4 基础质量工具**

- [ ] 配 `ruff`（lint+format）、`mypy`（类型检查）、`pytest`
- [ ] GitHub Actions：push 时自动 lint + test
- [ ] 配 `pre-commit` 钩子

### ✅ 验收标准

1. `cp .env.example .env`，填入 Key 后 `docker compose up -d` 一键启动
2. 访问 `http://localhost:8000/health` 返回 `{"status":"ok"}`
3. `docker compose exec backend python -c "from app.config import settings; print(settings)"` 正常输出
4. CI 通过

### ⚠️ 常见坑

- Windows/WSL 下 `pgvector` 镜像建议用 `pgvector/pgvector:pg16`，避免自行编译
- 飞书自定义机器人的"签名密钥"不是必填，但推荐开启；否则任何人知道 Webhook 都能发消息
- `.env` 一定加入 `.gitignore`，`.env.example` 不要写真 Key

## Step 1：MVP 核心链路打通（第 1-2 周）

### 🎯 目标

**最短链路**跑通：**采集 → LLM 摘要 → 飞书推送**，每天能收到一条消息即算成功。

此阶段一切从简：先用 SQLite 也行，先用同步调用也行，先 3 个信息源也够。

### 📦 交付物

- 能从 RSS 源抓取文章并入库
- 能调用 LLM 生成中文摘要 + 打分
- 每天 08:30 推送一条飞书卡片（含 TOP 5）
- 一个简单的 CLI 或 HTTP 接口可手动触发

### 📋 任务拆解

**1.1 数据模型 v1（只建必要表）**

- [ ] 用 SQLAlchemy 写 `Source`, `Article`, `Analysis`, `PushLog` 四张表
- [ ] 用 `alembic init` 初始化迁移，生成第一版 migration
- [ ] 写种子脚本 `scripts/seed_sources.py`，预置 5-10 个 RSS 源（OpenAI Blog、Anthropic News、Hugging Face Papers、机器之心、量子位等）

**1.2 采集器 —— 先只做 RSS**

- [ ] 定义抽象基类 `BaseCollector`（`fetch() -> List[RawArticle]`）
- [ ] 实现 `RSSCollector`（用 `feedparser`）
- [ ] 实现**去重**：以 `url` 为唯一键，配合 SHA1(正文) 作二次判断
- [ ] 实现**正文抽取**：`trafilatura.extract(html)`，失败降级到 RSS 的 `summary`
- [ ] 写 pytest 用例：mock 一个 RSS 源 XML，断言能解析出 N 条

**1.3 LLM 分析器**

- [ ] 封装 `LLMClient`（基于 `litellm`，便于切换模型）
- [ ] 写 Prompt 模板（直接复用需求文档附录 B），**强制 JSON 输出**
- [ ] 实现 `Analyzer.analyze(article) -> Analysis`，包含摘要/标签/分类/评分
- [ ] 加重试（`tenacity`）与 JSON 校验失败重试
- [ ] 加简单的 token 计数与成本估算日志

**1.4 飞书推送器**

- [ ] 实现 `FeishuNotifier`：用 `httpx` 发 Webhook
- [ ] 写飞书**交互卡片**模板（`msg_type=interactive`），支持：
  - 标题（含评分 emoji）
  - 摘要（300 字内）
  - 标签
  - 原文链接按钮
- [ ] 实现签名计算（若启用了签名）
- [ ] 推送前检查 `PushLog`，保证幂等（同一文章同一通道不重发）

**1.5 调度**

- [ ] 用 Celery Beat 配两个任务：
  - 每 30 分钟：`tasks.collect_all`
  - 每天 08:30：`tasks.send_daily_digest`
- [ ] `collect_all` 流程：抓取 → 入库 → 对新文章触发 `analyze_article.delay(id)`
- [ ] `send_daily_digest` 流程：查过去 24h 评分 ≥ 7 的 TOP 5，拼一张卡片推送

**1.6 极简管理接口**

- [ ] 加几个 REST 端点用于自测：
  - `POST /sources` 增加信息源
  - `POST /collect/{source_id}` 手动触发抓取
  - `GET /articles?min_score=7&limit=20` 查询
  - `POST /push/daily` 手动触发日报
- [ ] `GET /health` + `GET /stats`（总数、今日数、成本估算）

### ✅ 验收标准

1. 连续 3 天每天自动收到一条飞书卡片（TOP 5）
2. DB 中积累 ≥ 50 条文章，且有 ≥ 20 条的 `analysis` 记录
3. 手动 `POST /push/daily` 可立即触发推送，内容符合预期
4. 整体 LLM 调用成本日 < $0.5（若超支检查 Prompt 和文章数）

### ⚠️ 常见坑

- LLM 返回的 JSON 常常**前后带 ```json 包裹**或偶尔丢字段，一定要做**解析容错 + 重试**
- `feedparser` 对部分站点（尤其中文）编码识别差，必要时手动 `response.encoding = 'utf-8'`
- 飞书卡片单条消息长度有限制（约 30KB），摘要要截断
- Celery + 中文路径在 Windows 下容易诡异报错，建议**全程 Docker / WSL** 开发

## Step 2：数据强化与管理后台（第 3-4 周）

### 🎯 目标

- 把存储从 SQLite 升级到 **PostgreSQL + pgvector**
- 新增多种采集源（API 型）
- 构建 **Web 管理后台**，支持列表、筛选、搜索、手动操作
- 引入**即时推送**机制

### 📦 交付物

- 可用的 Web Dashboard（Next.js）
- 新增至少 3 种采集源类型
- pgvector 向量存储 + 基于向量的相似去重
- 即时高优推送通道跑通

### 📋 任务拆解

**2.1 数据库升级**

- [ ] `docker-compose.yml` 切到 `pgvector/pgvector:pg16`
- [ ] Alembic 加迁移：
  - 启用 `vector` 扩展
  - 给 `analysis` 加 `embedding vector(1024)`（或模型对应维度）
  - 给 `article.content` 建 GIN/`tsvector` 索引（或 `pg_trgm`）
- [ ] 写迁移脚本把旧 SQLite 数据导入 PG（若需要）

**2.2 Embedding 与语义去重**

- [ ] 选模型：默认 `bge-m3`（本地 1024 维）或 OpenAI `text-embedding-3-small`
- [ ] 分析阶段同时生成 embedding 存入 `analysis.embedding`
- [ ] 入库前先查 "近 7 天 + 余弦相似度 > 0.92" 的文章，判为重复则合并或丢弃

**2.3 更多采集源**

实现下列 Collector（每个独立 PR 验证）：

- [ ] `ArxivCollector`（调用 Arxiv API，按 cs.AI/cs.CL/cs.LG 每日拉取）
- [ ] `GitHubTrendingCollector`（爬 github.com/trending 或用第三方 API，筛 AI 相关 topic）
- [ ] `HackerNewsCollector`（用 Algolia API 按关键词 `AI/LLM/OpenAI` 过滤）
- [ ] `HTMLCollector`（通用爬虫，按 YAML 配置 CSS/XPath 规则，用于无 RSS 的博客）
- [ ] （可选）`RSSHubCollector`（自建 RSSHub 实例对接公众号/Twitter）

**2.4 即时推送**

- [ ] 修改 `analyze_article` 任务：分析完成后若 `score >= INSTANT_PUSH_MIN_SCORE`（默认 9），立刻入即时推送队列
- [ ] 即时推送 worker 每 5 分钟批量合并发送（避免洗版）
- [ ] 实现**静默时段**：22:00–07:30 的即时推送延迟到次日日报
- [ ] 卡片标题加 🔥 前缀与"突发"标识

**2.5 Web Dashboard**

- [ ] 用 `create-next-app` 初始化，装 `tailwindcss` + `shadcn/ui` + `tanstack-query` + `axios`
- [ ] 基础布局：侧栏（Dashboard / 文章 / 信息源 / 推送历史 / 设置）
- [ ] 登录：单用户，简单口令 + JWT（不做复杂权限）
- [ ] 页面：
  - **文章列表**：分页 + 标签/分类/评分/时间 筛选 + 关键词搜索
  - **文章详情**：显示原文链接、摘要、分析、评分理由、用户笔记
  - **信息源管理**：CRUD + "立即抓取"按钮 + 最近一次抓取状态
  - **推送历史**：按日期/通道/状态查询
  - **Dashboard 首页**：近 7 日文章数趋势、TOP 标签云、LLM 成本曲线
- [ ] 后端补齐对应 REST API（RESTful 风格，OpenAPI 文档自动生成）

**2.6 用户行为回流**

- [ ] 飞书卡片按钮（稍后读/不感兴趣/收藏）回调后端，更新 `user_action`
- [ ] Web 端支持标记 "收藏/隐藏/笔记"
- [ ] 所有行为作为后续偏好学习的训练信号

### ✅ 验收标准

1. Dashboard 上可完成：看文章 → 搜索 → 标记收藏 → 查看推送历史 全流程
2. 新增一个 HTML 爬虫源（YAML 配置），无需改代码
3. 突发重要文章（人工制造评分 9+）能在 5 分钟内推送到飞书
4. 相似度去重命中率可观察（日志中看到跳过记录）

### ⚠️ 常见坑

- pgvector 索引 `ivfflat` 在数据量少（<1万）时效果不佳，可先不建索引，暴力扫描即可
- Next.js + FastAPI 跨域，开发环境走 `next.config` 的 `rewrites` 代理
- Arxiv API 有 QPS 限制（3 秒一次），必须限速
- 通用 HTML 爬虫配 CSS/XPath 规则时，**一定要做页面 Hash 监控**，源站改版要能告警

## Step 3：智能化与用户体验（第 5-6 周）

### 🎯 目标

让系统**越用越聪明**：推荐更懂你、搜索更好用、报告更深度。

### 📦 交付物

- 偏好向量 + 个性化排序
- 语义搜索（自然语言问答式）
- 周报与趋势总结
- 成本与配额看板

### 📋 任务拆解

**3.1 偏好学习**

- [ ] 显式偏好：在设置页提供"方向 + 权重"配置（如 `Agent=0.9`）
- [ ] 隐式偏好：从 `user_action` 聚合（收藏 +1、稍后读 +0.3、不感兴趣 -0.5）
- [ ] 综合排序分 = `LLM评分 * 0.6 + 偏好匹配度 * 0.4`
- [ ] 每日日报按综合排序分选 TOP N

**3.2 语义搜索**

- [ ] Dashboard 搜索框支持两种模式：**关键词**（PG 全文） / **自然语言**（向量）
- [ ] 自然语言搜索：query → embedding → pgvector 余弦相似 TOP 20 → 可选 LLM rerank
- [ ] 加一个**问答式**入口："本周 Agent 领域有什么新进展？" → 系统召回相关文章 → LLM 汇总答复（附引用）

**3.3 周报与趋势**

- [ ] Celery Beat：每周一 09:00 触发 `weekly_digest`
- [ ] 流程：取过去 7 天全部 Analysis → 按标签聚合 → LLM 做**主题级总结**（每个主题 3-5 条要点 + 代表文章）
- [ ] 产出 Markdown 版 + 飞书多卡片（按主题分栏）+ 邮件 HTML

**3.4 深度解读（按需）**

- [ ] 对评分 ≥ 9 或用户标记"深读"的文章，触发 `deep_analysis`：
  - 全文翻译（如英文）
  - 关键技术点逐条拆解
  - 相关背景链接推荐（用向量检索历史文章）
  - 以 Markdown 存入 `analysis.deep_markdown`，Dashboard 单独 tab 展示

**3.5 成本与配额**

- [ ] 记录每次 LLM 调用的：`model / input_tokens / output_tokens / cost_usd`
- [ ] 实时汇总当月花费，接近预算 80% 时飞书告警，100% 时**暂停非关键分析**
- [ ] 提供"本地模型兜底"开关：超预算后自动切到 Ollama + Qwen2.5

### ✅ 验收标准

1. 用户设置"Agent=0.9"后，日报 TOP 5 明显偏向 Agent 类文章
2. 搜索框输入"推理效率优化"，能召回语义相关但不含该关键词的文章
3. 周报内容分主题、有深度，非简单 TOP 列表
4. 月度成本报表可看到按天 / 按任务类型的花费

### ⚠️ 常见坑

- "偏好权重"写死容易过拟合，建议加**最小探索率**（每天至少保留 20% 推送来自低匹配度文章，避免信息茧房）
- 周报 LLM 调用容易超长 context，要做**分主题分批汇总 + 二次合并**
- 本地模型效果与 GPT-4 有差距，切换时摘要字数/格式会抖动，Prompt 要加更严格的 few-shot 约束

## Step 4：稳定化、监控与多通道（第 7-8 周）

### 🎯 目标

系统**能放心跑半年不管**，同时扩展到更多使用场景。

### 📦 交付物

- 完整监控告警体系
- 多推送通道（邮件 / 企业微信 / Telegram 至少 +1）
- 本地模型 + 主备模型切换
- 部署文档 + 运维手册 + 备份方案

### 📋 任务拆解

**4.1 监控体系**

- [ ] Prometheus Exporter：采集/分析/推送任务的耗时、成功率、队列长度
- [ ] Grafana 看板：一屏看到"今日抓取数 / 今日分析数 / 推送数 / 失败率 / 花费"
- [ ] Loki/ELK：结构化日志（trace_id 贯穿采集 → 分析 → 推送）
- [ ] 告警规则：
  - 某源连续失败 ≥ 5 次
  - LLM 失败率 > 10%
  - 推送失败率 > 5%
  - 日花费 > 预算的 1/20

**4.2 多通道推送**

- [ ] 抽象 `BaseNotifier` 接口（`send(card) -> bool`）
- [ ] 至少实现 **邮件（SMTP + HTML 模板）** 或 **Telegram Bot**
- [ ] 用户配置每个推送事件（daily / instant / weekly）启用哪些通道

**4.3 本地模型支持**

- [ ] docker-compose 加一个 `ollama` 服务（可选，用 profile 控制）
- [ ] 在 `config` 加 `LLM_PRIMARY` / `LLM_FALLBACK`
- [ ] 主模型连续失败 3 次或月度超支后自动切换

**4.4 容灾与备份**

- [ ] 每日凌晨 3:00 触发 PG 逻辑备份（`pg_dump`）到 MinIO/本地
- [ ] 每周压缩归档，保留最近 8 周
- [ ] 编写一键恢复脚本 `scripts/restore.sh`

**4.5 部署与文档**

- [ ] 提供 `docker-compose.prod.yml`（加 Nginx、HTTPS、只读 DB 账号等）
- [ ] 写 `README.md`：30 秒看懂 + 5 分钟上手
- [ ] 写 `docs/运维手册.md`：日常巡检、常见故障、扩容方法
- [ ] 写 `docs/API.md`：由 FastAPI OpenAPI 自动生成 + 人工补充示例

**4.6 性能压测（可选）**

- [ ] 用 `locust` 模拟 1000 文章/天 的负载，观察各阶段耗时
- [ ] 必要时给 Analyzer 加并发池（尊重 LLM 并发配额）

### ✅ 验收标准

1. Grafana 上每个核心指标都有曲线
2. 人为 kill 掉 backend 容器，5 分钟内自愈并补跑任务
3. 月底查看报表：成本、文章数、推送数一目了然
4. 新同事按 README 30 分钟内跑起整套系统

### ⚠️ 常见坑

- `pg_dump` 大库时会锁，建议用 `--jobs` + `-Fd` 格式
- 邮件 HTML 在不同客户端渲染差异大，模板尽量用表格布局
- 监控本身也会挂：`alertmanager` 建议再推一份到飞书，独立链路

| Step 2 | PG 升级 + 1 个新采集源 | Web Dashboard（用 CLI 替代）、即时推送 |
| Step 3 | 偏好权重（纯配置式）| 语义搜索、周报深度总结 |
| Step 4 | 基础日志 + pg_dump | Prometheus/Grafana、多通道、本地模型 |

**结果**：4 周得到一个"能自动推送、数据可查、成本可控"的自用工具。

## 附录 B：每日开发节奏建议

```
09:00-09:30  看昨天的推送/数据，调整 Prompt 或参数（吃自己的狗粮）
09:30-12:00  主力编码（深度工作）
14:00-16:00  联调、测试、写单测
16:00-17:00  Code Review / Commit / 更新文档
17:00-18:00  跑一次端到端流程，观察今晚 & 明早的推送效果
```

**核心原则：每天至少让系统跑一次完整链路**，越早吃狗粮，越早发现 Prompt / 源质量 / 卡片样式的问题。

## 总结：推进时的"四问"

每完成一个子任务前，问自己：

1. **这个任务是当前 Step 的关键路径吗？**（不是就往后挪）
2. **最小实现是什么？**（MVP 先跑通，再优化）
3. **验收标准是什么？**（可度量、可观察）
4. **能否在今天内提交一次？**（保持小步快跑）

祝开发顺利。完成每个 Step 后记得打 Tag + 喝杯咖啡 ☕

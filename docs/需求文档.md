## 一、项目背景

### 1.1 背景描述

AI 技术发展日新月异，几乎每天都有新的论文、开源项目、产品发布、工程实践和行业动态出现。作为一名 AI 方向的高级开发人员，需要持续跟踪：

- 学术前沿（Arxiv、顶会论文）
- 开源社区（GitHub Trending、HuggingFace）
- 大厂技术博客（OpenAI、Anthropic、Google DeepMind、Meta AI、Mistral 等）
- 中文技术社区（机器之心、量子位、AI 前线、InfoQ 等）
- 社交媒体（Twitter/X 上的核心 KOL、知乎）
- 主流科技媒体（The Information、TechCrunch、VentureBeat 等）

手动浏览多个信息源耗时且容易遗漏。亟需一套**自动采集 → 智能分析 → 精准推送 → 持久存储**的系统，帮助用户以最低成本掌握行业最新动态。

### 1.2 项目目标

- **及时性**：核心信息延迟不超过 1 小时送达。
- **精准性**：通过大模型筛选与打分，每天推送不超过 20 条高价值信息（可配置）。
- **可读性**：提供中文摘要、关键词、重要性评级，1 分钟内可完成浏览决策。
- **可追溯**：所有采集内容持久化存储，支持全文检索、标签过滤和语义搜索。
- **低运维**：一键部署，自动运行，异常自愈。

### 1.3 非目标（Out of Scope）

- 不做内容原创/二次创作。
- 不做用户系统与多租户（v1.0 仅单用户使用）。
- 不做移动端 App（仅通过飞书/邮件/Web 访问）。

## 二、用户与场景

### 2.1 用户画像

| 属性 | 描述 |
|------|------|
| 角色 | AI 方向高级开发工程师 |
| 技术水平 | 熟悉 Python/Go/云原生，了解 LLM 与 RAG |
| 使用频次 | 每天 2-3 次（早间、午间、晚间） |
| 关注方向 | LLM、Agent、多模态、推理优化、RAG、AI Infra、应用落地 |
| 使用设备 | 飞书（手机 + PC）、浏览器 |

### 2.2 典型使用场景

**场景 1：早间资讯速览（8:30）**  
用户在通勤路上打开飞书，看到系统推送的"今日 AI 要闻 TOP 10"卡片，快速浏览摘要，对感兴趣的点击原文链接阅读。

**场景 2：技术深读（午休 / 晚间）**  
用户收到重要论文/项目推送，点击进入 Web 管理端，查看大模型生成的"深度解读"，标记收藏、添加个人笔记。

**场景 3：主题检索（周末）**  
用户回顾本周 "Agent 框架" 相关的所有资讯，通过 Web 端搜索标签/关键词/语义，快速梳理领域进展。

**场景 4：突发重要事件（任意时间）**  
如 OpenAI 发布新模型，系统识别为"高优先级"，立即（而非定时）推送到飞书并 @ 用户。

## 三、功能性需求

### 3.1 功能模块总览

```
┌─────────────────────────────────────────────────────────────┐
│                     AI 资讯聚合系统                           │
├───────────┬───────────┬────────────┬───────────┬────────────┤
│  采集模块  │  分析模块  │  存储模块   │  推送模块  │  管理后台   │
│  Collector│  Analyzer │  Storage   │  Notifier │  Dashboard │
└───────────┴───────────┴────────────┴───────────┴────────────┘
```

### 3.2 采集模块（Collector）

#### 3.2.1 信息源管理

- **F-COL-001** 支持配置多种信息源类型：
  - RSS / Atom Feed
  - 网页爬取（HTML 解析，基于 CSS/XPath 规则）
  - API 对接（Arxiv API、GitHub API、HackerNews API、Reddit API 等）
  - Twitter/X（通过官方 API 或第三方镜像）
  - 微信公众号（通过 RSSHub 等中转）
- **F-COL-002** 信息源配置存储于数据库，支持启用/禁用、修改抓取频率、设置优先级。
- **F-COL-003** 预置信息源清单（见附录 A），用户可自由增删。

#### 3.2.2 抓取调度

- **F-COL-004** 基于 Cron 的定时调度（默认每 30 分钟全量调度一次，单源频率可配置 5 分钟–24 小时）。
- **F-COL-005** 支持手动触发单源抓取。
- **F-COL-006** 抓取失败自动重试（指数退避，最多 3 次）；连续失败 5 次告警。
- **F-COL-007** 并发控制：单域名 QPS 限流，避免被封。

#### 3.2.3 去重与清洗

- **F-COL-008** 基于 URL + 内容哈希（SimHash/MinHash）去重，支持跨源去重。
- **F-COL-009** 正文提取：优先用 `readability`/`trafilatura` 去除广告导航。
- **F-COL-010** 原始 HTML 与清洗后文本均落盘保留。

### 3.3 分析模块（Analyzer）

#### 3.3.1 内容理解

- **F-ANA-001** 调用 LLM（默认 GPT-4o / Claude Sonnet，可切换本地模型）生成：
  - **中文摘要**（150–300 字）
  - **关键词/标签**（3–8 个，自动挂接标签体系）
  - **内容分类**（论文 / 开源项目 / 产品发布 / 融资 / 观点评论 / 教程 / 其他）
  - **重要性评分**（0–10 分，含评分理由）
- **F-ANA-002** 对长文（>4k token）进行分块摘要再汇总。
- **F-ANA-003** 支持对代码仓/论文做增强分析（如论文自动抽取 Method / Result / 创新点）。

#### 3.3.2 过滤与打分

- **F-ANA-004** 支持用户配置"兴趣偏好向量"（如"Agent=0.9, 硬件=0.3"），综合相关度重排序。
- **F-ANA-005** 推送阈值可配置（默认 ≥7 分推送，≥9 分触发即时推送）。
- **F-ANA-006** 支持黑名单关键词（过滤营销文、低质内容）。

#### 3.3.3 语义化与向量化

- **F-ANA-007** 为每条资讯生成向量 Embedding（默认 `bge-m3` 或 `text-embedding-3-large`），存入向量库用于检索与去重。

### 3.4 存储模块（Storage）

#### 3.4.1 数据模型

| 表 / 集合 | 关键字段 | 说明 |
|-----------|---------|------|
| `source` | id, name, type, url, cron, enabled, priority | 信息源 |
| `article` | id, source_id, url, title, author, published_at, fetched_at, raw_html, content, lang, hash | 原始文章 |
| `analysis` | id, article_id, summary_cn, tags, category, score, score_reason, embedding, model, cost | LLM 分析结果 |
| `push_log` | id, article_id, channel, status, pushed_at, error | 推送日志 |
| `user_action` | id, article_id, action (read/like/bookmark/hide), note, created_at | 用户行为 |
| `tag` | id, name, parent_id | 标签体系 |

#### 3.4.2 存储选型

- **F-STO-001** 关系型：PostgreSQL（主存储，含 JSONB）。
- **F-STO-002** 全文检索：PostgreSQL `pg_trgm` + `tsvector`，或可选 Elasticsearch/Meilisearch。
- **F-STO-003** 向量检索：`pgvector` 扩展，或可选 Milvus/Qdrant。
- **F-STO-004** 对象存储：原始 HTML / 图片存 MinIO 或本地文件系统。
- **F-STO-005** 缓存：Redis 用于任务队列（Celery/RQ）与幂等控制。

#### 3.4.3 数据保留策略

- **F-STO-006** 原始内容默认永久保留；超过 1 年的原始 HTML 可归档到冷存储。
- **F-STO-007** 支持手动/定时数据导出（JSON / CSV / Markdown）。

### 3.5 推送模块（Notifier）

#### 3.5.1 推送通道

- **F-NOT-001** **飞书（主通道）**：通过自建机器人 Webhook 或应用，发送富文本卡片（Interactive Card）。
- **F-NOT-002** **邮件（备通道）**：每日日报 HTML 邮件。
- **F-NOT-003** **预留扩展**：企业微信、Telegram、Slack、钉钉、Server 酱。

#### 3.5.2 推送策略

- **F-NOT-004** **日报**：每日 8:30 推送"今日 TOP N"卡片（N 默认 10，可配置）。
- **F-NOT-005** **即时推送**：评分 ≥9 的突发重要事件，5 分钟内推送。
- **F-NOT-006** **周报**：每周一 9:00 推送本周热点 + 领域趋势总结。
- **F-NOT-007** **静默时段**：默认 22:00–7:30 不推送即时消息，合并到次日日报。
- **F-NOT-008** 幂等保证：同一文章同一通道不重复推送。

#### 3.5.3 飞书卡片样式（示例）

```
┌──────────────────────────────────────────┐
│ 🔥 [9.2分] OpenAI 发布 GPT-5             │
│ 标签：#LLM #产品发布                      │
├──────────────────────────────────────────┤
│ 摘要：OpenAI 正式发布 GPT-5，支持 ... 在  │
│ 多项基准测试中超越 ... 支持原生 Agent ...  │
├──────────────────────────────────────────┤
│ 📅 2026-04-23  |  🌐 openai.com          │
│ [阅读原文]  [稍后读]  [不感兴趣]          │
└──────────────────────────────────────────┘
```

点击"稍后读/不感兴趣"回调后端，更新用户偏好。

### 3.6 管理后台（Dashboard）

- **F-DAS-001** Web UI（React/Next.js + Tailwind），支持：
  - 信息源管理（增删改查、测试抓取）
  - 文章列表（筛选、搜索、标签过滤、语义搜索）
  - 文章详情（原文 + 摘要 + 分析 + 笔记）
  - 推送历史与状态
  - 系统监控（抓取成功率、LLM 调用量与费用、存储占用）
  - 配置中心（推送阈值、偏好向量、API Key、静默时段）
- **F-DAS-002** 单用户登录（用户名 + 密码或飞书扫码）。

## 四、非功能性需求

### 4.1 性能

- 单次抓取周期（全部源）≤ 5 分钟。
- LLM 分析平均延迟 ≤ 15 秒/篇。
- Web 后台查询响应 ≤ 500ms（P95）。
- 支持单日处理 1000 篇文章（可横向扩展）。

### 4.2 可靠性

- 服务可用性 ≥ 99%（月度）。
- 抓取任务失败自动重试、告警。
- LLM 服务调用需熔断与降级（主模型不可用时切换备模型）。

### 4.3 成本控制

- **F-NFR-001** LLM 费用预算：月度不超过 $30（可配置上限，超额告警并暂停非关键分析）。
- **F-NFR-002** 对低分（<4 分）文章仅做摘要，不做深度分析，节省 token。
- **F-NFR-003** 支持切换为本地模型（Ollama + Qwen2.5/Llama3）以零成本运行。

### 4.4 安全与合规

- API Key、Webhook 等敏感信息加密存储（AES-256 或 Vault）。
- 爬虫遵守 `robots.txt`，设置合理 UA 与频率。
- 不抓取/推送付费墙内容（识别到则仅存链接与标题）。
- 日志脱敏，不记录完整 API Key。

### 4.5 可维护性

- 全链路日志（结构化 JSON，ELK 或 Loki）。
- Prometheus 指标 + Grafana 看板。
- 单元测试覆盖率 ≥ 70%。
- 所有配置通过环境变量 / YAML 文件管理。

### 4.6 可扩展性

- 模块间通过消息队列（Redis Stream / RabbitMQ）解耦。
- 新增信息源类型只需实现 `Collector` 接口。
- 新增推送通道只需实现 `Notifier` 接口。

## 七、验收标准

1. 能稳定运行 7×24 小时不少于 7 天，无人工干预。
2. 预置信息源抓取成功率 ≥ 95%。
3. 日报按时推送，格式美观，摘要准确率（人工抽检）≥ 90%。
4. Web 后台所有核心操作可用，搜索响应 < 1 秒。
5. 月度 LLM 成本在预算内。
6. 文档齐全：部署文档、API 文档、运维手册。

## 八、风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 信息源反爬 | 抓取失败 | User-Agent 池、代理池、降级为 RSSHub |
| LLM 费用超支 | 成本失控 | 预算熔断、分级分析、本地模型兜底 |
| 飞书 Webhook 限流 | 推送失败 | 批量合并、队列缓冲、重试 |
| 模型输出不稳定 | 摘要质量差 | Prompt 工程 + 输出结构化（JSON Schema）+ 校验重试 |
| 数据合规 | 版权风险 | 仅存标题/摘要/链接，不转载全文；尊重 robots.txt |
